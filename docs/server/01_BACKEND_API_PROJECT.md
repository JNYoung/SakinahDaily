# Backend API Project

Date: 2026-06-02

## Purpose

`services/backend-api` is the new Sakinah Daily backend BFF foundation. It gives
the app one backend-facing surface for:

- city search and curated city presets;
- timezone-aware prayer location resolution;
- generic content manifest/bundle delivery;
- local-only push preview and queue inspection.

The project is intentionally local and deterministic for MVP development. It
does not connect to live Supabase, Directus, FCM, APNs, OpenAI, or geocoding
providers.

## Product Decision

Timezone alone is not enough for prayer times. Prayer calculations need:

```text
latitude + longitude + timezone + calculation method
```

The backend should therefore return prayer-ready location records instead of a
timezone-only picker. A city search result must include coordinates, timezone,
country, and the recommended prayer calculation method.

## Local Architecture

```text
Flutter App
  -> /locations/cities
  -> /locations/resolve
  -> /manifest or /content/manifest
  -> /push/preview

services/backend-api
  ├── domain/locationCatalog.ts
  ├── domain/pushPreview.ts
  ├── repositories/fileContentPackStore.ts
  └── routes/*

services/content-agent
  -> writes .generated/content-pack/manifest.json
  -> writes .generated/content-pack/bundles/*.json
  -> writes .generated/content-pack/detail-bundles.json
```

## Endpoint Groups

### Locations

| Endpoint | Use |
|---|---|
| `GET /locations/cities` | Search/filter curated city catalog |
| `GET /locations/cities/:city_id` | Fetch one city |
| `POST /locations/resolve` | Resolve city ID or coordinates to a prayer-ready location |
| `GET /locations/timezones` | List catalog timezones by country or globally |

The MVP catalog includes high-priority Middle East and Indonesia cities. The
next backend step is replacing the in-code catalog with a reviewed database
table or imported city dataset.

### Content Delivery

| Endpoint | Use |
|---|---|
| `GET /manifest` | Existing generic client manifest path |
| `GET /bundles/:bundle_id.json` | Existing generic client bundle path |
| `GET /detail-bundle` | Existing generic client detail-bundle path |
| `GET /content/manifest` | Grouped BFF manifest path |
| `GET /content/bundles/:bundle_id.json` | Grouped BFF bundle path |
| `GET /content/detail-bundle` | Grouped BFF detail-bundle path |

The backend reads files generated by `services/content-agent`. It does not
generate or rewrite religious content.

### Push Preview

| Endpoint | Use |
|---|---|
| `POST /push/preview` | Validate and preview a local push payload |
| `POST /push/send` | Queue a local preview payload, still no FCM/APNs send |
| `GET /push/messages` | Inspect queued local payloads |
| `DELETE /push/messages` | Clear queued local payloads |

Push copy guardrails block cycle-sensitive lock-screen text, guaranteed outcome
claims, shaming tone, fatwa-like claims, and full Quran-like Arabic body text.

## Future Supabase Tables

Suggested first database-backed city table:

```sql
create table public.city_locations (
  id text primary key,
  label_en text not null,
  label_id text,
  label_ar text,
  country_code text not null,
  country_name_en text not null,
  country_name_id text,
  country_name_ar text,
  latitude double precision not null,
  longitude double precision not null,
  timezone text not null,
  prayer_calculation_method text not null,
  aliases text[] not null default '{}',
  source_label text not null,
  status text not null default 'published',
  review_status text not null default 'approved',
  updated_at timestamptz not null default now()
);
```

Public reads should be limited to:

```text
status = published
review_status = approved
```

The location table is public metadata. It must not become a location-history
store.

## Future Adapter Order

1. Keep local city catalog and content-pack file store green.
2. Add a Supabase city repository behind the same `LocationRoute` contract.
3. Add Directus/Supabase content repository or proxy behind the same manifest
   and bundle endpoints.
4. Add FCM dry-run job planning only after notification token storage and
   privacy review.
5. Add real push sending only after store/privacy review and lock-screen QA.

## Flutter Location Integration

The Flutter client can use the mock city catalog by enabling the Backend API
Dart defines:

```sh
flutter run \
  --dart-define=SAKINAH_BACKEND_API_ENABLED=true \
  --dart-define=SAKINAH_BACKEND_API_BASE_URL=http://127.0.0.1:8800
```

For Android emulator, use the host bridge URL:

```sh
flutter run -d emulator-5554 \
  --dart-define=SAKINAH_BACKEND_API_ENABLED=true \
  --dart-define=SAKINAH_BACKEND_API_BASE_URL=http://10.0.2.2:8800
```

For a physical Android device connected over USB, keep the app URL as
`http://127.0.0.1:8800` and reverse the port first:

```sh
adb reverse tcp:8800 tcp:8800
flutter run -d <device_id> \
  --dart-define=SAKINAH_BACKEND_API_ENABLED=true \
  --dart-define=SAKINAH_BACKEND_API_BASE_URL=http://127.0.0.1:8800
```

When the backend is disabled or unavailable, the Prayer location page falls
back to bundled city presets.
